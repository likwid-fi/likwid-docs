---
title: "还款"
description: "Likwid v2.2 的还款通过管理器原生的 repay 与 close 流程处理，不存在额外的合成代币赎回步骤。"
---

Likwid v2.2 的还款完全通过保证金管理器自身的状态迁移完成。用户在降低债务时 **不需要** 赎回某个外部合成代币。相反，债务、抵押物和镜像敞口都会直接在 vault 与 margin position 状态中更新。

协议为用户提供了两种主要的减仓或退出方式：**Repay** 与 **Close Position**。

**Close Position（平仓）**

这种方式允许用户通过池子自身的储备引擎部分或全部平掉仓位。

* **流程**：
  1. 用户发起平仓请求。
  2. 管理器计算目标平仓比例，并根据当前池子状态校验此次平仓是否可执行。
  3. 确认后，协议执行必要的储备更新与 swap 路径。
  4. 获得的代币用于偿还未结清债务。
  5. 清偿债务后若仍有剩余抵押物，则返还给用户。
* **需要注意**：
  * 这种方式会受到当前储备深度和费率条件影响。
  * 在高压力环境下，动态费率会提高大额平仓的成本。

#### 直接还款

这种方式下，用户直接用借入的资产偿还部分或全部债务，从而在不完全内部平仓的情况下为仓位降杠杆。

* **流程**：
  1. 用户通过 repay 流程向协议提供借入资产。
  2. 协议将这笔资产计入未偿债务。
  3. 根据剩余债务情况，系统可以部分释放抵押物，或者等待后续再完全平仓。
* **需要注意**：
  * 这种方式要求用户手中已经持有借入资产。
  * 当用户希望逐步降杠杆，而不是一次性让协议完全内部平仓时，这种方式更有用。

这两条路径共同为用户提供了足够灵活性：可以逐步降杠杆、一次性退出，也可以先把仓位调整到更适合后续完全平仓的状态。
