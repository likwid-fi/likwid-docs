---
title: "合成记账头寸"
description: "在 Likwid v2.2 中，y′ 与 x′ 更适合作为镜像敞口与债务状态的内部记账符号，而不是独立的外部代币。"
---

Likwid 早期文档中使用过 `y′` 或 `x′` 来描述杠杆仓位中的镜像一侧。在当前 v2.2 实现中，这些符号 **并不对应一个单独的、用户可见的 ERC-20 代币**。相反，协议通过 vault 与仓位管理器中的内部状态来记录合成敞口。

当本页提到 `y′` 或 `x′` 时，应将其理解为对以下 v2.2 机制的简写：

* **Mirror reserves（镜像储备）**：vault 在真实储备和 lend 储备之外，还维护一套镜像侧记账，用于表示借入或镜像敞口。
* **保证金仓位状态**：`LikwidMarginPosition` 会记录每个仓位的抵押物、债务以及累计记账值。
* **仓位 NFT 所有权**：用户对某个未平仓保证金头寸的索取权，通过 NFT 表示，而不是通过可自由流通的合成代币表示。
* **Balance-delta 结算**：开仓、还款、平仓和清算都会通过明确的余额变化来更新池子状态，而不是通过铸造某个外部记账代币来完成。

这种做法保留了 `y′` 在解释协议数学时的概念价值，同时避免让人误以为 Likwid v2.2 依赖某个外部执行模块或外部流通的合成代币合约。

对用户来说，结论很简单：协议仍然会记录杠杆债务和镜像敞口，但这些记录存在于 vault 与管理器合约内部，而不是存在于一枚单独的 token 中。
